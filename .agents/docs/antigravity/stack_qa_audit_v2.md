# 🕵️‍♂️ QA Audit Report [PHASE 2]: Глубинный Стресс-Тест Архитектуры

Первый аудит закрыл самые "глупые" ошибки (Rate Limits, Merge Conflicts, Базы данных). Теперь, когда наш артефакт обновлен, я прогнал систему через сценарии Enterprise-уровня. 

Даже с учетом новых правок, я нашел **4 скрытые угрозы**, которые могут парализовать работу రోя (Swarm) или нанести финансовый ущерб.

---

### 🚨 Угроза №1: Коллизия Локальных Баз при E2E (Race Conditions)
* **Сценарий поломки:** QA-инженеры (5 потоков Jules) запускают тесты `Playwright` одновременно в локальном окружении `localhost:3000`. Поток 1 регистрирует юзера `admin@test.app`. Через миллисекунду Поток 2 пытается зарегистрировать того же юзера. Тест Потока 2 падает. Агенты начинают "чинить тест", не понимая, что они делят одну БД.
* **Инженерный Фикс (Изоляция Тест-Тенантов):** 
  - Каждый QA-поток обязан изолировать свое рабочее пространство.
  - Либо агент поднимает уникальный Docker-контейнер с чистой БД на лету (`--project-name qa-worker-X` и порт `3001, 3002...`).
  - Либо использует In-Memory SQLite изолированно на процесс теста, чтобы E2E моки никогда не пересекались по сети.

### 🚨 Угроза №2: Интеграционный Паралич Стейт-Менеджеров (State Mismatch)
* **Сценарий поломки:** *Google Stitch* сгенерировал роскошную корзину товаров и чекаут, используя локальные хуки React (`useState`). Наш проект использует глобальный стор `Redux` или `Zustand`. Frontend-Агент отрефакторил CSS (как мы решили в 1-м аудите), но корзина работает "в вакууме" и не влияет на счётчик в хедере.
* **Инженерный Фикс (State Lifting Policy):**
  - При пост-процессинге выдачи от *Google Stitch*, Frontend-агент обязан не только "покрасить" классы, но и прочитать Контракт API (Этап 0). 
  - Агент обязан "поднять состояние" (Lift State Up) — удалить все `useState` и привязать сгенерированный UI к глобальным селекторам хранилища проекта (Zustand/Redux). Сборка не проходит линтинг, пока UI-компонент не свяжется с глобальным стором.

### 🚨 Угроза №3: Разорение на Токенах (API Billing Runaway)
* **Сценарий поломки:** Воркер ловит неочевидный баг. Он переписывает код -> генерирует интерфейс в Stitch -> тест снова падает -> он переписывает код. Jules попадает в Infinite Retry Loop (бесконечную петлю) в 3 часа ночи. За 5 часов 100 агентов сжигают $5,000 на вызовах Claude/Nanobana/Stitch, генерируя мусор.
* **Инженерный Фикс (Circuit Breaker & Hard Budgets):**
  - "Предохранитель бюджета". В `State.json` Техлид вшивает хардлимит: *Максимум 5 итераций фиксов (Dead Letter Threshold) ИЛИ максимум $5 API Token Budget на одну Jira-задачу*.
  - При пробое любого лимита Jules намертво замораживает задачу, отправляет алерт-сообщение Вам: "Лимит исчерпан. Либо баг нерешаемый, либо алгоритм сбоит. Разрешите ручное вмешательство".

### 🚨 Угроза №4: Кровавая Забастовка Безопасника (False Positives Blockade)
* **Сценарий поломки:** Воркер *Security Auditor* читает код. SAST (сканер уязвимостей) ругается на 50 ложных срабатываний (False Positives) — например, тестовый `JWT_SECRET=123` в мок-папке. Security Agent блокирует Pull Request, так как "Система не безопасна". QA и Dev агенты пытаются переписать тесты, чтобы угодить SAST сканеру, руша весь проект.
* **Инженерный Фикс (Audit-Only & Override Rules):**
  - Security Agent переводится в режим `Audit-Only (Warning)` для папок с тестами, моками и документацией. Жёсткое применение паттернов безопасности (`Enforcing`) работает только в директориях `src/auth`, `src/payments` и ядре API.
  - У Техлида (и у Вас) есть право `Override` (игнорирования) некритичных ложных срабатываний SAST-сканера по конкретным меткам комментариями в PR. Разум превалирует над паранойей.
